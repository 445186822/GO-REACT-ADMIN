  │ Nivo       │ 13k+  │ 原生 React        │ 颜值高、SVG 渲染           │
  ├────────────┼───────┼───────────────────┼────────────────────────────┤
  │ visx       │ 19k+  │ 原生 React        │ Airbnb 出品、高度可定制    │
  └────────────┴───────┴───────────────────┴────────────────────────────┘

  推荐：echarts-for-react + echarts

  理由：
  1. 三个参考项目全部用 ECharts，你团队已有经验
  2. 图表类型最全（柱状图、折线图、饼图、仪表盘、热力图、桑基图…）
  3. 中文文档完善，百度维护
  4. echarts-for-react 是 React 官方推荐封装（2.9k stars）
  5. 支持 dark 主题，配合你的主题系统

  安装：npm install echarts echarts-for-react

  四、架构层面问题

  1. BasicLayout 过于庞大（669 行）

  内联了 AppearanceDrawer（160 行）、5 个菜单辅助函数、WebSocket 逻辑。应拆分为：
  layouts/
  ├── BasicLayout.tsx       (~200行 主框架)
  ├── AppearanceDrawer.tsx  (独立)
  ├── layoutHelpers.ts      (纯函数)
  └── useTabNavigation.ts   (标签逻辑 hook)

  2. 权限粒度不足

  多个模块（Scheduler、RecycleBin、DataDict）所有 CRUD 操作复用 settings:update。每个模块应有独立的 CRUD 权限码。

  3. 缺少 Error Boundary

  全项目 0 个 ErrorBoundary。任何一个页面渲染异常 →整个 Layout 白屏。应该在 BasicLayout 和每个 lazy 路由外层包
  ErrorBoundary。

  4. 导出反模式

  FileCenter、UserList 等页面通过 page_size: 10000 一次性拉全量数据再导出。数据量大时直接 OOM。应该改为后端流式导出。

  5. WebSocket 连接重复

  BasicLayout（通知 badge）和 NotificationCenterPage 各开一个 WS 连接到同一端点。应提取为 useNotificationSocket() hook。
